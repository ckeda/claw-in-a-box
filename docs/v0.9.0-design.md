# v0.9.0 “Face” — design note

Status: **owner review required; no implementation or deployment is authorized by this note.**

v0.9.0 builds on `agent/v0.8.1`. It exposes persisted operational state to the
Console without changing the byte-stable NANDA free-tier contract, Pay-to-Claim,
strict mode, or verdict semantics. All work is staged first at
`test.clawinabox.xyz` with `DISCOVERY=off`. `console.clawinabox.xyz` remains
frozen until the owner explicitly approves deployment after judging.

## 1. Access model and common behavior

The server, not the Console, enforces three roles:

| Role | Credential | Allowed v0.9 data/actions |
|---|---|---|
| Visitor | none | existing public Console, aggregate-only metrics, recovery challenge issuance/submission |
| Agent owner | `X-Agent-Secret` | that agent's spend view, existing strict toggle and Telegram rebinding |
| Operator | `Authorization: Bearer …` | approval list/god-view; future global admin actions only |

`OPERATOR_BEARER_KEY` is a high-entropy host-managed env value. The server
SHA-256 hashes both the supplied and configured values into equal-length buffers
and uses `crypto.timingSafeEqual`; it never logs or persists the key. Missing
configuration makes operator-only endpoints return `503 feature_disabled`.
Missing, malformed, or incorrect bearer credentials return the same `401
unauthorized` response with `WWW-Authenticate: Bearer`.

v0.9 adds no global mutation endpoint. The operator key therefore gates only
`GET /v1/approvals` in this version and is reserved for later global/admin
mutations. It is deliberately **not** a bypass for agent-secret checks,
Pay-to-Claim, strict mode, Telegram webhook authentication, or a human approval.

All three new read APIs query MySQL, never memory fallback, and return `503
feature_disabled` unless `PERSISTENCE=on`, DB connected, and hydration complete.
Sensitive responses use `Cache-Control: no-store`. Existing CORS gains
`authorization` in allow-headers and exposes `Retry-After` plus the informational
rate-limit headers.

## 2. Read-only APIs

### 2.1 Operator approval feed

```http
GET /v1/approvals?status=pending&limit=25
Authorization: Bearer <operator key>
```

- `status` is optional and one of `pending|approved|denied|expired`.
- `limit` defaults to 25, must be an integer from 1 through 100, and results are
  newest first. No agent-secret access is accepted because a list is a god-view.
- The feed reads the existing `approvals` table and therefore follows its
  existing approximately 30-minute retention/sweep window.

```json
{
  "approvals": [
    {
      "approval_id": "9f2c1a...",
      "status": "pending",
      "final_verdict": null,
      "request": {
        "agent_id": "agent-7",
        "action": "spend",
        "amount": 150,
        "destination": "merchant-x"
      },
      "created_at": "2026-07-19T09:00:00.000Z",
      "resolved_at": null
    }
  ],
  "count": 1
}
```

Invalid filters return `400 invalid_query`; invalid operator auth returns 401.
The existing knower-visible `GET /v1/approvals/:id` remains unchanged.

### 2.2 Agent-owner spend view

```http
GET /v1/agents/agent-7/spend
X-Agent-Secret: <that agent's current secret>
```

The path id is the authorization scope. An unclaimed id returns 404; a missing,
wrong, or another agent's secret returns 403. The response never includes wallet,
destination, policy, Telegram chat, or secret material.

```json
{
  "agent_id": "agent-7",
  "day": "2026-07-19",
  "spent_today": 31,
  "history": [
    {
      "id": "1842",
      "delta": -1,
      "spent_after": 30,
      "reason": "verdict_expired_refund",
      "ref_id": "4f86e1...",
      "created_at": "2026-07-19T09:05:00.000Z"
    }
  ],
  "history_scope": "v0.9_forward_last_50"
}
```

`daily_spend` remains the authoritative current-day snapshot. A new append-only
table records v0.9-forward ledger changes for the last-50 history:

```sql
CREATE TABLE spend_ledger (
  id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  agent_id VARCHAR(191) NOT NULL,
  day CHAR(10) NOT NULL,
  delta DOUBLE NOT NULL,
  spent_after DOUBLE NOT NULL,
  reason VARCHAR(32) NOT NULL,
  ref_id VARCHAR(64) NULL,
  created_at TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  INDEX spend_agent_created (agent_id, created_at)
);
```

Rows are written only when the budget ledger changes: automated allow,
human-approved bound verdict, and bound-verdict expiry refund. They contain no
destination or policy. Enforcement stays memory-first as in v0.8; the history is
observational/eventually consistent and is not used to authorize spending.

### 2.3 Public aggregate metrics

```http
GET /v1/metrics
```

```json
{
  "generated_at": "2026-07-19T09:05:00.000Z",
  "window_seconds": 86400,
  "agents": { "claimed": 12, "strict": 4 },
  "approvals": { "pending": 2, "approved_24h": 7, "denied_24h": 3, "expired_24h": 1 },
  "verdicts": { "pending": 5, "consumed_24h": 21, "expired_24h": 4 },
  "spend": { "active_agents_today": 8, "ledger_changes_24h": 29 }
}
```

The response is built from count-only aggregate SQL and cached for 15 seconds:
agent counts come from `agents`, pending counts from live tables, 24-hour counts
from typed `events`, and today's active count from `daily_spend`. It offers no
filters or grouping and must never serialize agent ids, wallets, destinations,
chat ids, event payloads, secrets, individual amounts, or public revenue totals.
This also limits differencing attacks against small populations.

## 3. Wallet-signature secret recovery

One route has two explicit phases. Both are separately rate-limited in addition
to the normal free-route IP limit and fail closed without a hydrated DB.

### 3.1 Issue challenge

```http
POST /v1/agents/recover
Content-Type: application/json

{"agent_id":"agent-7"}
```

```json
{
  "agent_id": "agent-7",
  "nonce": "<32-byte base64url>",
  "message": "Claw-in-a-Box agent secret recovery\nVersion: 1\nDomain: test.clawinabox.xyz\nAgent ID: agent-7\nNonce: ...\nIssued At: ...\nExpires At: ...\nStatement: Rotate the secret for this agent. This does not authorize a payment.",
  "expires_at": "2026-07-19T09:10:00.000Z"
}
```

The nonce is random, expires after five minutes, and the DB stores only its
SHA-256 hash. The exact message includes version, `API_HOST`, agent id, nonce,
issued/expiry times, and a non-payment statement. Domain separation prevents a
staging signature from being replayed on production. The response does not
reveal `claimed_by`.

### 3.2 Verify and rotate

```http
POST /v1/agents/recover
Content-Type: application/json

{"agent_id":"agent-7","nonce":"...","signature":"0x..."}
```

The server hashes the presented nonce, loads its row, reconstructs the canonical
message from the presented nonce plus stored domain/timestamps, performs EIP-191
`personal_sign` recovery, normalizes the recovered EVM address, and compares it
to the durable `agents.claimed_by`. On success, one DB transaction locks the
nonce and agent rows, verifies unused/unexpired state, writes a new secret hash,
marks the nonce used, and commits before updating the hydrated cache.

```json
{
  "agent_id": "agent-7",
  "agent_secret": "<new one-time 32-byte base64url secret>",
  "recovered_at": "2026-07-19T09:06:00.000Z"
}
```

The old secret becomes invalid immediately; the new secret is shown once and
the response is `no-store`. Expected errors are 400 malformed input/signature,
403 signer mismatch, 404 unclaimed id, 409 nonce already used, 410 expired
nonce, 429 rate limit, and 503 persistence unavailable. Only EOA/EIP-191 wallets
are in the proposed v0.9 scope; contract-wallet EIP-1271 is an open question.

```sql
CREATE TABLE agent_recovery_nonces (
  nonce_hash CHAR(64) PRIMARY KEY,
  agent_id VARCHAR(191) NOT NULL,
  domain VARCHAR(255) NOT NULL,
  issued_at_ms BIGINT UNSIGNED NOT NULL,
  expires_at_ms BIGINT UNSIGNED NOT NULL,
  used_at TIMESTAMP(3) NULL,
  INDEX recovery_agent_issued (agent_id, issued_at_ms),
  INDEX recovery_expiry (expires_at_ms)
);
```

Issuance and successful recovery emit `recovery_challenge_issued` and
`agent_secret_recovered`; rejected submissions emit a rate-bounded
`agent_recovery_failed` reason code. Audit payloads never contain nonce,
signature, plaintext secret, raw IP, or full wallet address.

Replay defenses: cryptographically random nonce, exact canonical message,
five-minute expiry, environment/domain binding, atomic `used_at` transition,
and a transaction lock ensure concurrent reuse yields exactly one success.
Issuing a new challenge does not invalidate another active challenge, avoiding
an unauthenticated nonce-invalidation DoS. Expired rows are swept later.

## 4. Console v0.9 (repository only)

The static SPA adds a credentials/settings surface with three slots in
localStorage: visitor (no credential), one agent-owner `{agent_id, secret}`, and
one operator bearer key. Values are masked, never placed in query strings,
analytics, logs, rendered JSON, or error messages, and have per-slot clear plus
“clear all” controls. The requested localStorage model has inherent XSS risk, so
the Console keeps no third-party runtime scripts, uses text rendering rather
than raw HTML, and ships a restrictive static CSP.

- Approvals: operator key enables the list feed; without it, existing single-id
  lookup remains usable.
- Spend: agent-owner slot calls only its matching `/v1/agents/:id/spend` and can
  drive the existing strict toggle and authenticated Telegram rebinding.
- Metrics: public aggregate cards; no credential required.
- Recovery: injected-wallet `personal_sign` flow; the Console never requests or
  handles a private key.
- API client: add only the explicit v0.9 free endpoints to its safety allowlist;
  `/paid/*` and `/paid-okx/*` remain blocked.
- Environment: use a build-time API base, with repository/staging work targeting
  `https://test.clawinabox.xyz`. No build is uploaded to
  `console.clawinabox.xyz` until owner approval after judging.

The Console source currently lives in the `agent/console-build-week` history,
not in the v0.8.1 ancestry from which this branch was required to be cut. The
owner must choose the history-preserving integration method before Step 2.

## 6. Backlog fixes included in v0.9

1. **Claim duplicate race:** normalize MySQL duplicate-key forms (`ER_DUP_ENTRY`,
   errno 1062 / integrity SQLSTATE) to `409 already_claimed` without marking the
   DB disconnected. A losing concurrent claim must fail before settlement, so
   exactly one caller receives 201 and exactly one payment settles.
2. **Informational rate-limit headers:** every response that passes through the
   existing free-route limiter receives `RateLimit-Limit`,
   `RateLimit-Remaining`, and `RateLimit-Reset` (delta seconds). Existing 429
   behavior and `Retry-After` remain authoritative and unchanged. No response
   body changes.

## 7. Threat notes

- Operator key compromise exposes the approval god-view; it must remain only in
  host env/owner localStorage, never a frontend build value or repository file.
- Agent secrets remain bearer credentials. Recovery rotates rather than reveals
  the stored secret and cannot bypass the durable settlement wallet anchor.
- A public recovery challenge endpoint permits nuisance issuance and claimed-id
  enumeration (already possible via duplicate claim); per-IP and per-agent
  limits, independent active nonces, bounded TTL, and cleanup limit the DoS.
- Nonce/message/signature values are sensitive authentication artifacts and are
  redacted from logs/audit. Transactional one-time use closes replay and races.
- Public metrics use a fixed aggregate schema, no filters, and a short cache to
  prevent identity inference and DB-amplification queries.
- Spend history is private and intentionally excludes destinations; the
  operator approval feed may contain them because it is explicitly god-view.
- No new endpoint changes the five original NANDA response bodies or adds auth
  to unclaimed/non-strict free use.

## 8. Test and staging acceptance list

- Operator auth: env missing 503; missing/malformed/wrong bearer identical 401;
  correct key works; constant-time hash comparison; agent secret cannot list.
- Approval list: status/limit validation, newest-first ordering, retention,
  empty results, no memory fallback, and SQL-injection-safe parameters.
- Spend: own secret works; missing/wrong/cross-agent secret 403; unclaimed 404;
  charge/refund ledger rows and last-50 order; no private fields leak.
- Metrics: unauthenticated aggregate schema, fixed keys only, 15-second cache,
  no identifier/payload leakage, and DB-unavailable 503.
- Recovery: issue/sign/succeed; old secret revoked/new secret works; wrong wallet,
  tampered message, malformed signature, expiry, single-use, concurrent replay,
  restart survival, environment replay, rate limits, audits, and DB-down 503.
- Claim race: concurrent first claims produce one 201 + one 409 and one settle on
  both payment rails; no 500 and no secret in the losing response.
- Rate limits/CORS: informational headers on success, `Retry-After` on 429,
  browser exposure, and `authorization` allowed.
- Console: credential headers scoped correctly; localStorage clear/redaction;
  visitor fallback; list/spend/metrics/recovery UI; paid-route safety block.
- Root `llms.txt` snapshot/content check; no automated upload.
- Full existing suite plus new tests on Node 18 and Node 22. NANDA five-endpoint
  structures remain byte-stable; claim and strict semantics regress unchanged.
- Staging at `test.clawinabox.xyz` with `DISCOVERY=off`, separate DB/test bot,
  controlled claimed wallet recovery, restart nonce survival, and no mainnet or
  frozen-Console deployment.

## 9. Owner decisions required before implementation

1. **Console history integration:** should Step 2 cherry-pick the existing
   `agent/console-build-week` commits onto `agent/v0.9.0` (recommended, preserves
   the Codex/feedback audit trail), or merge that branch?
2. **Approval retention:** is the existing approximately 30-minute live feed
   sufficient for v0.9, or must the list reconstruct longer history from audit
   events/new storage?
3. **Spend history:** approve the new PII-minimized `spend_ledger`, v0.9-forward
   scope, fixed last-50 response, and retention policy (proposed: 90 days)?
4. **Recovery wallet scope:** EOA/EIP-191 only for v0.9 (recommended), or is
   EIP-1271 smart-contract-wallet verification required now?
5. **Recovery throttles:** approve proposed limits of 5 challenge issues per
   hour per IP, 3 per hour per agent, and 10 signature submissions per hour per
   IP, all with `Retry-After`.
6. **Operator key rotation:** one active `OPERATOR_BEARER_KEY` only, or support a
   temporary previous key during rotation? Recommendation: one active key and a
   planned brief Console interruption for the smaller attack surface.
7. **Operator mutations:** confirm v0.9 adds no operator approval override or
   other global mutation (recommended); bearer auth gates the list now and
   reserves the boundary for later admin APIs.
